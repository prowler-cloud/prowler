Before writing any code for a new provider, pick one of the three scopes below. Each has its own required artifacts, registrations, and completion criteria, so choosing early avoids doing work twice.

<Note>
This page answers **"how far does this provider need to reach?"** (CLI-only, external package, or full Prowler Cloud/Local Server support). It is a different question from [Classifying your Provider](./provider#classifying-your-provider) in the full provider guide, which answers **"what kind of provider is it?"** (SDK, API, Tool/Wrapper, or Hybrid). Decide your scope here first, then use that guide to classify the provider type.
</Note>

## Choose Your Scope

<Columns cols={3}>
  <Card title="Built-in CLI/SDK Provider" icon="terminal">
    Ships inside this repository under `prowler/providers/`. Usable from the Prowler CLI. This is what every current provider (AWS, Azure, NHN, Vercel, and the rest) is today.
  </Card>
  <Card title="External Entry-Point Provider" icon="puzzle-piece">
    Distributed as a separate, independently installable Python package. Discovered by Prowler at runtime through a `prowler.providers` entry point, without living in this repository.
  </Card>
  <Card title="Full-Stack Provider" icon="layer-group">
    A built-in provider that also integrates with the Prowler API and UI, so it can be used from Prowler Cloud or Prowler Local Server, not only the CLI.
  </Card>

## Scope 1: Built-in CLI/SDK Provider

**Checklist:**

1. Classify the provider type (SDK / API / Tool / Hybrid) and whether it is regional, using [Classifying your Provider](./provider#classifying-your-provider).
2. Create `prowler/providers/<name>/` following the structure in the `prowler-provider` skill (`skills/prowler-provider/SKILL.md`) and [Step 1: Create the Provider Backend](./provider#step-1-create-the-provider-backend-cli-integration) in the full guide.
3. **Discovery is automatic.** `Provider.get_available_providers()` (`prowler/providers/common/provider.py`) finds any package under `prowler/providers/` (other than `common`) whose `<name>_provider.py` defines a `<Name>Provider` class inheriting from `Provider`. No manual registration is required just for the class to exist and be importable.
4. **CLI help text is also automatic for new providers.** `prowler/lib/cli/parser.py` keeps a hardcoded `known_providers` set for formatting reasons, but any provider not in that set is appended to the CLI usage/help text automatically, using a `_cli_help_text` class attribute on your provider class (empty string if unset). Set `_cli_help_text` on your provider class to get a real description instead of a blank one — you do not need to edit `parser.py` itself.
5. **Argument parsing and instantiation are still manual today.** Every current provider is wired into the `elif` chain in `init_global_provider()` (`prowler/providers/common/provider.py`) to parse its provider-specific CLI arguments and instantiate itself correctly. The base `Provider` class exposes a `from_cli_args()` override point intended to let a provider handle this itself instead of extending that chain, but as of this writing no built-in provider implements it — treat it as an available option, not the established pattern to copy.
6. Add SDK unit tests (see the `prowler-test-sdk` skill and [Unit Testing](./unit-testing)).
7. **Completion criteria:** `prowler <name> --help` lists the provider with a real description; `prowler <name> -h` shows its arguments; SDK tests pass.

<Warning>
Registrations for a single provider can live in several places and drift out of sync with each other — for example, `prowler/lib/cli/parser.py`'s `known_providers` set currently includes `llm`, while the `Provider` enum in `prowler/config/config.py` does not. Don't assume one file's list is authoritative; check discovery (`prowler/providers/common/provider.py`) and the checklist above instead.
</Warning>

<Note>
The `Provider` enum in `prowler/config/config.py` (`Step 12: Add to Config` in the full guide) is not referenced anywhere else in the codebase as of this writing. It may be relied on by tooling this page doesn't cover, so following the full guide's Step 12 is still reasonable, but don't treat a missing entry there as broken unless you've confirmed a specific consumer needs it.
</Note>

## Scope 2: External Entry-Point Provider

**Checklist:**

1. Same provider-class and service/check structure as a built-in provider (Scope 1, steps 1-2), packaged in your own installable Python project instead of `prowler/providers/`.
2. Register your provider class under the `prowler.providers` entry-point group in your package's `pyproject.toml` (or equivalent), matching how `prowler/providers/common/provider.py` discovers entry points via `importlib.metadata.entry_points(group="prowler.providers")`.
3. A built-in provider with the same name always takes precedence over an external entry-point provider.
4. **Completion criteria:** with your package installed alongside Prowler, `prowler <name> --help` and a real scan both work, without any change to the Prowler repository itself.

<Note>
There is currently no first-party reference implementation of this path in this repository to point to directly — NHN Cloud, the closest community-maintained example mentioned elsewhere in the docs, ships in-tree as a built-in provider rather than as an external entry point. The discovery code supports this scope, but test end to end rather than assuming parity with the built-in path.
</Note>

## Scope 3: Full-Stack Provider (SDK + API + UI)

**Checklist:**

1. Complete Scope 1 in full first.
2. Follow [Step 2: Integrate the Provider in the API](./provider#step-2-integrate-the-provider-in-the-api) — includes the `ProviderChoices` enum, serializers, views, the credentials model, and API tests.
3. UI integration: the dedicated UI section of the full provider guide is not yet written (tracked as its own phase of [issue #12026](https://github.com/prowler-cloud/prowler/issues/12026)). Until it lands, use a recently added full-stack provider (for example Okta or Vercel) as a working reference for the UI-side changes: provider type registration, credential forms/schemas, icons, and labels.
4. **Completion criteria:** see the CLI/API/UI subsections of [Checklist for New Providers](./provider#checklist-for-new-providers) in the full guide.
