---
title: 'Getting Started with Okta'
---

This guide covers setting up Okta security scanning with Prowler. The Okta provider authenticates with **OAuth 2.0 (private-key JWT)** as a service application — no end-user login, fully read-only by scope.

<Note>
**Initial release scope**

The initial Okta provider release ships with one check (`signon_global_session_idle_timeout_15min`, mapped to DISA STIG V-273186). Additional services and checks land in subsequent releases.
</Note>

## Prerequisites

- An Okta organization. The UI examples below use **Identity Engine** naming such as **Global Session Policy**; Classic Engine exposes the equivalent sign-on policy concepts under older names.
- A **Super Administrator** account on that organization to perform the one-time service-app setup.
- Python 3.10+ and Prowler 5.26.0 or later installed locally.

## 1. Create the Okta service app

1. In the Okta Admin Console, go to **Applications → Applications → Create App Integration**.
2. Pick **API Services** as the sign-in method, name it (for example, `Prowler Scanner`), and create the app.
3. On the new app's **General** tab, switch **Client authentication** to **Public key / Private key**.
4. Either upload an existing public JWK or click **Add key** to let Okta generate one. Download / copy the **private key** material (PEM or JWK) — Okta only shows it once.

<Warning>
**Disable the "Public client app admins" org setting**

Under **Settings → Account**, ensure **Public client app admins** is **off**. When enabled, newly created service apps can be auto-assigned the Super Admin role after scopes are granted, which would silently grant Prowler far more privilege than required.
</Warning>

## 2. Grant the required OAuth scopes

In the service app's **Okta API Scopes** tab, click **Grant** on every scope Prowler needs. For the initial release only one scope is strictly required:

- `okta.policies.read`

The full list of scopes used by future checks (grant any you want pre-provisioned):

| Scope | Used by |
|---|---|
| `okta.policies.read` | Sign-on / password / authentication policies |
| `okta.apps.read` | Application sign-on policies |
| `okta.authenticators.read` | Smart Card, Okta Verify FIPS settings |
| `okta.idps.read` | Identity provider checks |
| `okta.networkZones.read` | Network zone checks |
| `okta.logStreams.read` | Log streaming checks |
| `okta.brands.read` | Sign-in widget / DOD banner check |
| `okta.users.read` | Future user-related checks |
| `okta.roles.read` | API token role checks |
| `okta.apiTokens.read` | API token checks |

## 3. Assign the Read-Only Administrator role

In **Admin assignments** for the service app, assign the built-in **Read-Only Administrator** role. This further constrains what the granted scopes can do at runtime, in line with least privilege.

## 4. Configure Prowler

Set the following environment variables. Using env vars is preferred over passing the values on the CLI.

```bash
export OKTA_ORG_DOMAIN="acme.okta.com"
export OKTA_CLIENT_ID="0oa1234567890abcdef"
export OKTA_PRIVATE_KEY_FILE="/secure/path/to/prowler-okta.pem"
# Optional — defaults to "okta.policies.read"
export OKTA_SCOPES="okta.policies.read"
```

The private key file may contain either a PEM-encoded RSA key or a JWK JSON document.

### Supplying the Private Key as Content

For automated environments where writing the key to disk is not desirable (Prowler App, CI runners, container secrets, etc.), the private key may be passed directly as a string:

```bash
export OKTA_ORG_DOMAIN="acme.okta.com"
export OKTA_CLIENT_ID="0oa1234567890abcdef"
export OKTA_PRIVATE_KEY="$(cat /secure/path/to/prowler-okta.pem)"
```

`OKTA_PRIVATE_KEY` takes precedence over `OKTA_PRIVATE_KEY_FILE` when both are set. The private key is intentionally not exposed as a CLI flag — secrets must be supplied via environment variables only.

## 5. Run a scan

```bash
# Run all Okta checks
poetry run python prowler-cli.py okta

# Run a specific check
poetry run python prowler-cli.py okta --check signon_global_session_idle_timeout_15min
```

Prowler will print a summary table; full findings are written to the configured output formats.

## Troubleshooting

### STIG Rule Ordering

The initial check is mapped to DISA STIG `V-273186` / `OKTA-APP-000020`. Prowler implements the STIG procedure as written: the **Default Policy** must have a **Priority 1** rule that is **not** `Default Rule`, and that rule must set **Maximum Okta global session idle time** to 15 minutes or less.

This is stricter than simply finding the same timeout value somewhere else in the policy set. A compliant custom rule in another policy, or a compliant timeout on the built-in `Default Rule`, doesn't satisfy this STIG procedure.

### Default Scopes

Prowler requests a fixed set of OAuth scopes on every token exchange. The default is a single scope that covers the bundled initial check:

- `okta.policies.read`

The service app must have that scope granted in the **Okta API Scopes** tab. When the granted set is narrower than the requested set, the token request fails with an `invalid_scope` error and the scan stops at provider initialization.

When additional checks are enabled — or when running against a service app that exposes a different scope set — override the default with `OKTA_SCOPES` (comma-separated string for the env var) or `--okta-scopes` (space-separated list for the CLI):

```bash
# Environment variable — comma-separated
export OKTA_SCOPES="okta.policies.read,okta.apps.read,okta.users.read"

# CLI flag — space-separated
poetry run python prowler-cli.py okta \
  --okta-scopes okta.policies.read okta.apps.read okta.users.read
```

For the full catalog of OAuth scopes exposed by the Okta Management API, refer to the [Okta OAuth 2.0 scopes documentation](https://developer.okta.com/docs/api/oauth2/).

<Note>
As new services and checks land in the Okta provider, the default scope list grows alongside them. Re-check the granted scopes on the service app after each Prowler upgrade and grant any newly required `okta.*.read` scopes in the Admin Console.
</Note>

### Common Errors

- **`OktaInvalidOrgDomainError`** — the org domain must be `<org>.okta.com` (or `.oktapreview.com` / `.okta-emea.com` / `.okta-gov.com`). Pass the bare hostname only — no `https://` scheme, no path, no trailing slash.
- **`OktaPrivateKeyFileError`** — confirm the file is readable and contains a non-empty PEM or JWK body.
- **`OktaInsufficientPermissionsError`** — the credential probe reached Okta but the service app cannot perform the request. The error string carries `invalid_scope`, `Forbidden`, `not authorized`, or `permission`. Fix by granting the missing `okta.*.read` scope from **Okta API Scopes** and confirming the **Read-Only Administrator** role is assigned to the service app.
- **`OktaInvalidCredentialsError`** — the credential probe reached Okta but Okta rejected the JWT. Typically the private key on disk does not match the public JWK uploaded to the service app, or the JWT signing parameters are wrong. Regenerate the keypair and re-upload the public JWK.
- **Token requests failing for an unknown scope** — the app was granted a narrower scope set than `OKTA_SCOPES` requests. Either narrow `OKTA_SCOPES` or grant the missing scopes in the Admin Console.
